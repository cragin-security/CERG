## 1. Purpose and Scope

Upstream CERG is an open-source cybersecurity operating model starter kit. It already solves structural problems most documentation-first approaches miss: the three-pillar model, the evidence-by-cadence discipline, the system-control profile, the adoption sequencing. What it does not yet define is how a consulting practice turns that model into repeatable client engagements across organizations that vary in size, regulatory burden, sector, and maturity.

This package provides that missing layer. It does not redefine CERG. It operationalizes CERG as a practice asset.

This plan applies to every consulting engagement in which CERG is the implemented or recommended operating model, regardless of client sector, team size, or regulatory regime.

> **What This Plan Does and Does Not Do**
>
> This plan defines how CERG is deployed: which engagement tier fits which client profile, which deliverables are standardized, what the handover must contain, and how practice knowledge is compressed and reused. It does not redefine CERG artifacts, override document owners, or substitute for professional legal, regulatory, or compliance advice. Every engagement still requires an independent applicability determination by qualified counsel, compliance leadership, and, where required, the client's auditors.

## 2. The CERG Consulting Value Proposition

Most clients arrive with one of three interchangeable complaints: policy exists but is not operational; compliance is performed as a separate exercise; or audit findings recur because the underlying operating model cannot sustain evidence production. These are structural failures, not control failures.

CERG solves the structure. The consulting value is accelerating the client's arrival at a running program.

The differentiated client outcomes a consulting engagement must produce:

1. **A signed, adapted policy spine installed in the client's repository within the first 30 days.** A policy that exists as a draft is not a program. A policy that has been adapted to the client's regulator set, organization name, scale tier, and executive sponsor, and has been signed, is a program foundation.
2. **A populated risk register with at least one completed review cycle by day 30.** This converts CERG from a document set into an operating rhythm.
3. **Evidence produced as a byproduct of normal work, not assembled before audits.** Clients who produce evidence quarterly from their runbooks pass audits; clients who reconstruct evidence annually do not.
4. **A defined handover that lets the client run without the consulting team present.** The consulting deliverable is not the client's dependency. It is the client's independence.

The engagements in this plan are designed to produce those four outcomes predictably.

## 3. Engagement Tier Model

Clients are mapped to one of four engagement tiers at intake. The tier determines scope, cadence, team composition, deliverables, and price anchor. Tier assignment is not a maturity judgment; it is a fit-for-delivery-model judgment.

### 3.1 Tier Definitions

| Tier | Client Profile | Typical Headcount | Typical Maturity | Engagement Model |
|---|---|---|---|---|
| **Tier 1: Foundations** | New security function, no operating model, minimal tooling | 1-15 security FTEs | Initial or Formative per CERG-GOV-MAT-001 | Fixed-scope, 60-90 day advisory engagement |
| **Tier 2: Structure** | Existing function with fragmented policies and ad hoc processes | 5-30 security FTEs | Defined or Managed | Scoped advisory and implementation support, 90-180 day engagement |
| **Tier 3: Compliance-to-Operations** | Regulated environment with recurring audit burden | 15-100+ security FTEs | Managed or Optimized | Multi-phase engagement; ongoing advisory cadence |
| **Tier 4: Strategic Partner** | Enterprise with multiple regulatory regimes; strategic security investment | 50-300+ security FTEs | Optimized and beyond | Retainer-based; co-design of operating model evolution |

### 3.2 Tier Feature Matrix

| Feature | Tier 1 | Tier 2 | Tier 3 | Tier 4 |
|---|---|---|---|---|
| Org profile build and token scheme | Included | Included | Included | Included |

| Standards gap analysis | Advisory only | Included | Included | Included |

| Regulatory package implementation | Out of scope | Out of scope | Included | Included |

## 4. Client Profile Routing

Before any scope is written, the client is classified across three independent axes: size, regulatory burden, and sector. The combination determines the default engagement tier and the mandatory overlay packages.

### 4.1 Size Classification

| Classification | Security Team Size | Org Headcount Range |
|---|---|---|
| **Small** | 1-5 FTEs | Under 500 |
| **Mid** | 5-30 FTEs | 500 to 10,000 |
| **Large** | 30-150 FTEs | 10,000 to 100,000 |
| **Enterprise** | 150+ FTEs | 100,000+ |

The size classification does not override the engagement tier. A small team at a regulated enterprise may still be Tier 3 because the regulatory burden drives scope.

### 4.2 Regulatory Burden Classification

| Classification | In-Scope Frameworks |
|---|---|
| **None** | No external regulatory assessment burden; internal governance only |
| **Single** | One primary regulated framework (e.g., ISO 27001 only; CMMC only) |
| **Multiple** | Two or more concurrent regulated frameworks (e.g., CMMC CUI + NIST 800-53; CMMC CUI + SOX ITGC) |
| **Critical Infrastructure** | NERC-CIP registered entity; or sector-designated critical infrastructure with federal/state oversight |

The regulatory classification determines which operational packages must be selected and implemented.

### 4.3 Sector Classification

| Sector | Primary Risk Themes |
|---|---|
| **Technology / SaaS** | Identity, SaaS supply chain, SOC 2 evidence, AI intake |
| **Financial Services** | SOX ITGC, BCP, incident response, access recertification |
| **Healthcare / Life Sciences** | HIPAA or HITRUST, CUI, vendor risk, data classification |
| **Manufacturing / OT** | Asset inventory, segmentation, change control, supplier integrity |
| **Energy / Utilities** | NERC-CIP compliance, OT exposure management, BCP |
| **Defense Industrial Base** | CMMC L2, SPRS score visibility, POA&M discipline |
| **Government / Federal** | FedRAMP, FISMA mapped controls, IR interface, CUI |
| **Cross-sector / Holding** | Multiple regulators via subsidiary, centralized governance |

Sector does not change the engagement tier, but it triggers specific overlay modules and influences the illustrative examples in the rendered token profile.

Each row names the operational package, CUI/CMMC applicability, when it must be selected, when it may be deferred, and the consulting activity required at package activation.

### 5.1 Core Overlays

|---|---|---|---|---|
| [`CERG-PLN-CUI-001`](../plans/CERG-PLN-CUI-001_CUI_CMMC_Operational_Package.md) | Yes | Client pursues CMMC certification or handles CUI | Client has no current or planned CUI scope | Confirm deferred status in risk register; set target date |
| [`CERG-PLN-CIP-001`](../plans/CERG-PLN-CIP-001_NERC_CIP_Operational_Package.md) | No | Client is a NERC-CIP registered entity | Client does not operate BES Cyber Assets | Confirm deferred status in risk register |
| [`CERG-PLN-SOX-001`](../plans/CERG-PLN-SOX-001_SOX_ITGC_Operational_Package.md) | No | Client is publicly traded or SOX-bound by contract | Client is private with no SOX exposure | Confirm deferred status in risk register |
| [`CERG-PLN-ISO-001`](../plans/CERG-PLN-ISO-001_ISO_IEC_27001_Operational_Package.md) | Context-dependent | Client pursues or sustains ISO 27001 certification | Client has no external certification obligation | Confirm deferred status in risk register |
| [`CERG-PLN-PRIV-001`](../plans/CERG-PLN-PRIV-001_Privacy_and_Data_Protection_Operational_Package.md) | Context-dependent | Client handles personal data under GDPR, CCPA, or analogous regime | Client has no PII or sensitive personal data scope | Confirm deferred status in risk register |

### 5.2 Sector-Specific Overlays

| Sector | Mandatory Enhancers | Priority Standards |
|---|---|---|
| **Technology / SaaS** | Vendor security questionnaire enhancement; SaaS evidence collection checklist | STD-IT-001, STD-AC-001, STD-LM-001, PRC-TPRM-001 |
| **Financial Services** | SOX ITGC package; access recertification cadence; BCP testing | PLN-SOX-001, STD-RES-001, STD-AC-001, PLN-BC-001 |
| **Healthcare / Life Sciences** | Privacy operational package; CUI handling; vendor risk | PLN-PRIV-001, STD-CUI-001, PRC-TPRM-001 |
| **Manufacturing / OT** | OT visibility decision matrix; change control plant-floor variants | STD-OT-001, STD-AC-001, PRC-AR-001 OT variant |
| **Energy / Utilities** | Full NERC-CIP operational package; OT exposure SLAs | PLN-CIP-001, STD-OT-001, STD-CFG-001 |

A deferred overlay is not a silent gap. Per [`CERG-GOV-IMP-001`](../governance/CERG-GOV-IMP-001_Implementation_and_Adaptation_Guide.md) §7.2, each deferred standard or package is recorded in the risk register with owner, compensating controls, and target adoption date.

### 6.2 Reusable Artifacts Owned by the Practice

The consulting practice maintains version-controlled copies of the following artifacts. They are the practice's curriculum and asset library. Do not recreate these per engagement.

## 7. Engagement Workflow by Tier

Workflow is the sequence of consulting activities for each tier. It replaces hand-waving delivery with an auditable engagement path.

### 7.1 Tier 1: Foundations Workflow

| Phase | Activity | Duration | Output |
|---|---|---|---|
| **Intake** | Client profile questionnaire; tier confirmation; conflict-of-interest review | Day 1 | Tier assignment and scope acknowledgment |
| **Profile** | Build `cerg-org-profile.yml`; validate against checklist; client signs off token values | Days 2-5 | Signed org profile file |
| **Render and Adapt** | Run render tool; review output for naming, scale, sector correctness; document client-specific exceptions | Days 5-8 | Rendered corpus with exception log |
| **Catalog and Sign-Off** | Update CAT-001 for adoption decisions; Governance pillar leader and CISO both sign policy | Days 8-12 | Signed adoption record |
| **Run** | Load risk register; produce first 10 entries; facilitate first risk review meeting | Days 13-25 | First completed risk review |
| **Validate** | Gate 2-to-3 readiness check per IMP-005 §6; verify MVC produced cadence outputs | Days 26-30 | Readiness memo |
| **Handover** | Practice exit memo; handover meeting; improvement register seeded | Days 28-30 | Handover package |

Total target duration: 30 days from signed engagement.

### 7.2 Tier 2: Structure Workflow

Extends Tier 1 with:

- Standards gap analysis and layered adoption recommendation
- Architecture review procedure activation and first two projects through intake
- Exposure management cycle activation with first SLA report
- Access management standard adoption
- Regulatory overlay selection and system-to-package mapping
- First metrics dashboard
- Second maturity assessment at day 90

Total target duration: 90 days for structural advisory phase; optional ongoing advisory at monthly cadence.

### 7.3 Tier 3: Compliance-to-Operations Workflow

Extends Tier 2 with:

- Regulatory operational package full implementation
- System Security Profile per in-scope system
- Evidence library activation and auditor-facing evidence package template
- Quarterly testing cadence activation
- Deficiency workflow live test and playbook
- IR adjacent function handoff confirmation
- Third maturity assessment plus recommendations for Tier 4 transition

Total target duration: 180-day implementation phase; ongoing advisory at quarterly cadence.

### 7.4 Tier 4: Strategic Partner Workflow

- Co-design of operating model evolution roadmap
- Multiple regulatory overlay governance design
- Workforce architecture and role-to-person mapping support
- New standard or procedure design through practitioner guild
- Practitioner build-out and interview guide use
- Succession planning and talent review support

Ongoing retainer engagement; quarterly strategic reviews.

## 8. Practice Operating Requirements

### 8.1 Practitioner Competency

Every practitioner delivering CERG engagements must demonstrate:

- Practical experience running a CERG-structured security program or completing the framework walkthrough with a review by a CERG-certified practitioner
- Comfort with the CERG corpus, catalog, render tool, and validator
- Awareness of which artifacts are fixed content and which are adaptable variables
- Specific regulatory literacy for any package they are implementing; CUI/CMMC requires a different literacy profile than SOX ITGC

A practitioner who cannot explain the difference between a `PRC` and a `PLN` should not be advising clients on adoption decisions.

### 8.2 Independence Requirement

The consulting engagements in this plan are advisory, not product sales. The practice selects rate-limiting tools and vendors with client input; the model works with the client's existing stack. When a client's tooling is insufficient for a procedure, the practice recommends specific capability categories and documents the gap in the improvement register; it does not prescribe a vendor.

### 8.3 Licensing and Attribution

CERG is open source. The practice contributes back to the upstream repository. When a customer-identified fix or improvement is merged into upstream CERG, the client is credited and their investment becomes a public asset.

Practice-side reports reference upstream CERG artifacts by Document ID; they do not reproduce controlled content unnecessarily. The client's forked and adapted repository is the authoritative implementation; practice reports are commentary around it.

## 9. Knowledge Management and Asset Reuse

The practice compresses what it learns engaging clients into reusable assets. This section defines the protocol.

### 9.1 Post-Engagement Debrief

Every tier 2 and above engagement produces a structured debrief within five business days of handover completion. The debrief covers:

- What the client already had that was better than the base CERG model
- Where CERG's utility-first narrative created friction for the client's sector
- Which regulatory package needed the most tailoring
- Which validator or tooling gaps surfaced most often
- What the practice would change about the engagement design for the same client profile next time
- Named practitioner observations about adoption blockers and accelerators

The debrief is archived practice-side. It is not shared with the client without explicit consent.

### 9.2 Improvement Feedback into CERG

Structural and generalized findings feed upstream as:

- Issues on the upstream CERG repository, categorized by severity (against the existing 2026 issues catalog structure)
- Pull requests for validator gaps, template omissions, documentation clarifications, or missing overlays
- ADR-style practice notes for reference by future practitioners

The practice maintains a prioritized practice backlog derived from client engagement findings. Quarterly practice reviews look at which backlog items should become upstream contributions.

### 9.3 Client Profile Register

The practice maintains a running register of client profile types it has served, key characteristics of each profile type, and the engagement design that worked. This is the practice's competitive differentiation: faster and more accurate tier assignment, better engagement design, and fewer surprises.

The register is anonymized. It tracks:

- Profile type (from Section 4 classification)
- Tier assigned
- Which overlay packages were selected and required tailoring
- Time to first risk review
- Day-90 maturity delta
- Client satisfaction and referral patterns

## 10. Risk and Liability Boundaries

### 10.1 What the Practice Does Not Accept

The practice does not accept liability for the client's regulatory certification outcome. CERG is a framework, not a guarantee. The practice designs processes, adapts artifacts, trains the client's team, and supplies repeatable deliverables. The client retains responsibility for:

- Legal applicability determinations
- Auditor engagement and assessment scope
- Control implementation and testing
- Regulatory notification decisions and timelines
- Third-party vendor selection and contracts

Where the practice identifies a regulatory gap or deficiency, it is surfaced, documented in the improvement register, and assigned an owner. It is not resolved by consulting presence; it requires the client's executive and operational action.

### 10.2 External Interface and IR Seam

Per [`CERG-GOV-OM-001`](../governance/CERG-GOV-OM-001_CERG_Operating_Model.md), IR is an adjacent function. The practice confirms the IR adjacent seam during Tier 3 engagements: the incident response playbook exists, is owned by the standing IR team, and names the CERG record that is updated after each incident. The practice does not own the IR function; it verifies the handshake.

### 10.3 Data Handling

Client data in the engagement follows the client's own data classification scheme, or falls under the practice's standard consulting confidentiality agreement. The rendered CERG corpus produced for a client remains the client's asset. Practice templates do not embed client data.

## 11. Measuring Success

### 11.1 Per-Engagement Success Indicators

| Indicator | Measurement |
|---|---|
| Policy signed | Day 30 milestone in Tier 1; day 30 metric for all tiers |
| Risk register populated and reviewed | Number of entries populated; review conducted |
| First cadence cycle completed | Governing artifacts produced; dated records in repository |
| Day-90 maturity delta | Maturity score delta from baseline at 90 days |
| Deferred decisions recorded | Every deferred standard or package captured in risk register |
| Client self-reports reduced audit prep time | Subjective measure at engagement close |
| Improvement register implemented | Open items with owner and target date |
| Handover accepted | Client sign-off on exit memo |

### 11.2 Practice-Level Success Indicators

| Indicator | Target |
|---|---|
| Tier assignment accuracy | Engagements run on or ahead of planned duration |
| Upstream contribution rate | At least one upstream issue or PR per quarter |
| Client repeat engagement rate | Tier 2-4 clients convert to ongoing advisory |
| Client referral rate | Referrals from satisfied clients within 12 months |
| Artifact reuse rate | Percentage of engagements using at least 80% of standardized deliverables without custom creation |
